# Cosmoria Folder Structure

This document explains the structure of the Cosmoria repository and the responsibility of each directory.

Cosmoria is designed as a modular backend engine built with Go, focused on multi-tenant SaaS applications.

---

cosmoria/

# 🚀 cmd/cosmoria/

Entry point and CLI.

```
cmd/cosmoria/
  main.go    # Entry point — calls cli.Run()
  cli.go     # CLI command dispatch (serve, dev, migrate, init)
  dev.go     # Hot-reload watcher via fsnotify
```

**Commands:**

| Command | Description |
|---------|-------------|
| `cosmoria serve` | Start the server (default) |
| `cosmoria dev` | Hot reload — watch .go files, rebuild & restart |
| `cosmoria init` | Generate .env, docker-compose.yml, Dockerfile |
| `cosmoria migrate new <name>` | Create migration pair |
| `cosmoria migrate up/down` | Run/revert migrations |

# 🧠 internal/

Core business logic. Cannot be imported externally.

### internal/app/

```
internal/app/
  server.go    # BuildHandler: wires all services → handlers → routes → middleware
```

### internal/api/

HTTP layer:
- `router.go` — Router wrapping `http.ServeMux`
- `handlers/` — Request handlers (health, auth, admin, tenants, rbac, collections, records, files, ws, apikeys, audit)
- `middleware/` — Middleware chain (logging, auth, admin auth, tenant, rbac)

### internal/auth/

SaaS user authentication: signup, login, JWT, bcrypt, API keys (apikeys.go).

### internal/adminauth/

Platform admin authentication: admin_users, admin_project_roles, separate JWT.

### internal/configfile/

Declarative YAML configuration:
- `config.go` — Types matching YAML structure
- `parser.go` — YAML file parser
- `applier.go` — Applies config to DB (idempotent per name)

### internal/collections/

Dynamic schema engine: Collection/Schema/Field models, CRUD, ValidateData.

### internal/records/

Records CRUD with cursor-based pagination, schema validation via collections.

### internal/core/

Engine bootstrap:
- `config.go` — env-based config with zero-config defaults
- `app.go` — App struct with Run / Shutdown

### internal/db/

Database layer: PostgreSQL connection pool (`pgx/v5`), migrations (`golang-migrate`).

### internal/tenant/

Multi-tenancy: tenant service (CRUD, user assignment), context helpers.

### internal/rbac/

Role-Based Access Control: roles, permissions, user assignment, CheckAccess.

### internal/storage/

File storage abstraction layer with two backends:
- `backend.go` — `StorageBackend` interface + factory `NewBackend(cfg)` with auto-detection
- `s3.go` — `S3Client` for S3-compatible storage (minio-compatible API)
- `local.go` — `LocalBackend` writing to `<STORAGE_PATH>/<key>` (default `./data/files`)
- `service.go` — Files CRUD (upload, get, list, delete) with DB metadata + backend operations
- `uploader.go` — Multipart form parser for file uploads

### internal/audit/

Audit logging system:
- `logger.go` — Async logger (`Log()` runs in goroutine, never blocks)
- `service.go` — Cursor-paginated list endpoint for audit entries

### internal/realtime/

Real-time system with PostgreSQL `LISTEN`/`NOTIFY`:
- `hub.go` — Client registry and event fan-out by project + tenant
- `pubsub.go` — Publisher (pg_notify) + Subscriber (dedicated PG conn, dynamic channels)
- `client.go` — Gorilla WebSocket per-connection: read/write pumps, ping/pong
- `events.go` — JSON-serializable event + WebSocket message types

### internal/mcp/

MCP server (Model Context Protocol):
- `server.go` — JSON-RPC 2.0 message loop over stdin/stdout
- `tools.go` — Tool definitions and handler dispatch (~21 tools)
- `types.go` — JSON-RPC request/response types and server state machine

# 📦 sdk/

### sdk/typescript/

TypeScript SDK with zero runtime dependencies:

```
sdk/typescript/
  package.json    # @cosmoria/sdk metadata
  client.ts       # CosmoriaClient class with typed methods for all endpoints
  index.ts        # Re-exports client + types
  api.ts          # Generated by openapi-typescript (regenerate via scripts/generate-sdk.sh)
```

Usage:

```typescript
import { CosmoriaClient } from "./sdk/typescript/index.ts";
const client = new CosmoriaClient("http://localhost:8080");
const auth = await client.auth.signup("user@ex.com", "pass", "proj-id");
client.setToken(auth.token);
```

# 🎯 Design Principles

Cosmoria follows these architectural rules:

- **Multi-tenancy mandatory**: every data query includes `project_id` + `tenant_id`
- **No global state**: explicit dependency injection everywhere
- **Minimal dependencies**: stdlib preferred, avoid frameworks
- **OpenAPI-first**: all endpoints documented via swaggo annotations, spec served at runtime
- **CLI-first**: control via `cosmoria` commands, zero env vars required for dev
- **Separation of concerns**: each `internal/` module has a strict responsibility
- **PostgreSQL-first**: JSONB for flexible schemas, migrations for structured data
